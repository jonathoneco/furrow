#### `work/orchestration-delegation-contract`
**Branch:** `work/orchestration-delegation-contract` | **Sessions:** ~5 | **Conflict risk:** none (independent)

**Theme:** the orchestration/delegation contract treated as one design with three faces — schema, split, enforcement — shipped together so the contract is real (defined + located + constrained), not a convention.

**Contract trinity** (the three todos that together form the contract):
- `handoff-prompt-artifact-template` — **layer interface (the schema):** the 7-field handoff artifact (grounding / persona / goals / constraints / enforcement / deliverables / EOS-report) is what crosses the layer boundary in both directions
- `furrow-context-isolation-layer` — **layer split (the side):** which side of the boundary is furrow-aware (orchestrator) vs furrow-unaware (specialist), and how delegation flows
- `delegation-boundary-enforcement` — **layer constraint (the gate):** what is allowed to cross and what must not; prevents silent boundary erosion via prompt scoping, tool-permission scoping, and a test that runs furrow-unaware specialists on non-Furrow projects to assert zero leakage

**Supporting infrastructure** (consumed by the contract):
- `context-routing-infrastructure` — per-step context loading rules; the contract uses these to populate the handoff artifact
- `design-pattern-context-construction` — the meta layer that ratifies the approach
- `standardize-artifact-presentation` — how artifacts (including the handoff itself) are surfaced in conversation

**Sequence:** lock the schema first (`handoff-prompt-artifact-template`) — it's the API. Then the split (`furrow-context-isolation-layer`) anchors which side does what. Then the enforcement (`delegation-boundary-enforcement`) makes the contract real, not aspirational. Supporting infrastructure ships in parallel.
