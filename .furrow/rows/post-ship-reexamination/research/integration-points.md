The CLI only writes `roadmap.yaml`; the optional `.furrow/almanac/roadmap.md` companion is written by the agent per `commands/triage.md:213-219`.

**Insertion point for "Active Observations"**:

- In `roadmap.yaml`: add a new top-level key `active_observations` alongside `deferred` (between `phases` at `bin/alm:755` and `deferred` at `bin/alm:756`). `alm triage` reads `.furrow/almanac/observations.yaml`, filters by computed-active predicate, and emits `[{id, title, trigger, fired_at}]`.

**Parsing-consumer risk**: low. `commands/triage.md:38-45` specifies the three parsed keys (`todos`, `graph`, `conflicts`) ‚Äî adding a new key is additive and doesn't break jq filters that name specific keys. `cmd_next` (`bin/alm:772+`) reads `roadmap.yaml` via `yq -o=json`; it queries specific paths (`.phases[]`, `.phases[].number`, etc., see `bin/alm:803-806`), so adding a sibling key `active_observations` is safe. `alm validate` (which `cmd_triage` gates at `bin/alm:523`) currently validates todos.yaml only ‚Äî D3 will need a parallel `validate-observations` step but that's out of scope for this research.
