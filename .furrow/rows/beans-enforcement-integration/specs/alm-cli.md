# Implementation Spec: `alm` CLI

**Deliverable:** alm-cli (D5)
**File:** `bin/alm` (~800 lines estimated)
**Specialist:** cli-designer

---

## 1. Architecture

Monolithic POSIX shell script with `case`-based subcommand dispatch. No subshell sourcing, no plugin architecture — a single file with function-per-subcommand.

### Dependencies

| Tool | Purpose | Required |
|------|---------|----------|
| `yq` | YAML read/write | yes |
| `jq` | JSON manipulation, formatting | yes |
| `python3` | Topological sort (Kahn's algorithm), schema validation | yes |
| `awk` | Text extraction (summary.md sections) | yes (POSIX) |

### Data Directory

All almanac data lives under `.furrow/almanac/`:

```
.furrow/almanac/
  todos.yaml          # TODO registry (schema-validated)
  roadmap.yaml        # Structured roadmap (generated by alm triage)
  rationale.yaml      # Component rationale (future)
```

The `ALM_DIR` variable at the top of the script controls this path:

```sh
ALM_DIR=".furrow/almanac"
ALM_TODOS="$ALM_DIR/todos.yaml"
ALM_ROADMAP="$ALM_DIR/roadmap.yaml"
```

### Row Artifact Paths

Extraction sources use the `.furrow/rows/` directory:

```sh
ROWS_DIR=".furrow/rows"
```

### Error Handling

Standard `die()` function prints to stderr with `alm:` prefix and exits non-zero:

```sh
die() { printf 'alm: %s\n' "$1" >&2; exit "${2:-1}"; }
```

### Help / Usage

`alm --help` and `alm` (no args) print usage listing all subcommands with one-line descriptions. Each subcommand supports `alm <cmd> --help` for detailed usage.

## 2. Script Skeleton

```sh
#!/bin/sh
set -eu

ALM_DIR=".furrow/almanac"
ALM_TODOS="$ALM_DIR/todos.yaml"
ALM_ROADMAP="$ALM_DIR/roadmap.yaml"
ROWS_DIR=".furrow/rows"
SCHEMA_FILE="adapters/shared/schemas/todos.schema.yaml"

die() { printf 'alm: %s\n' "$1" >&2; exit "${2:-1}"; }

# --- subcommand functions ---
cmd_add()      { ... }
cmd_extract()  { ... }
cmd_list()     { ... }
cmd_show()     { ... }
cmd_triage()   { ... }
cmd_next()     { ... }
cmd_validate() { ... }
cmd_render()   { ... }
cmd_help()     { ... }

# --- dispatch ---
cmd="${1:-help}"
shift 2>/dev/null || true

case "$cmd" in
  add)      cmd_add "$@" ;;
  extract)  cmd_extract "$@" ;;
  list)     cmd_list "$@" ;;
  show)     cmd_show "$@" ;;
  triage)   cmd_triage "$@" ;;
  next)     cmd_next "$@" ;;
  validate) cmd_validate "$@" ;;
  render)   cmd_render "$@" ;;
  help|--help|-h) cmd_help ;;
  *)        die "unknown command: $cmd" ;;
esac
```

## 3. Subcommands

### 3.1 `alm add`

**Purpose:** Create a new TODO entry in `todos.yaml`.

**Interface:**

```
alm add --title T --context C --work W [--risks R] [--refs R1 R2 ...]
         [--source-type TYPE] [--source-work-unit NAME] [--seed-id ID]
         [--depends-on D1 D2 ...] [--files-touched F1 F2 ...]
```

**Required flags:** `--title`, `--context`, `--work`

**Behavior:**

1. Validate required flags present; die with usage if missing.
2. Generate `id` as kebab-case slug from `--title`:
   - Lowercase, replace non-alphanumeric with `-`, collapse runs, trim leading/trailing `-`.
   - Truncate to 50 characters at word boundary.
   - If slug collision exists in `todos.yaml`, append `-2`, `-3`, etc.
3. Set `source_type` to `"manual"` unless `--source-type` overrides.
4. Set `created_at` and `updated_at` to current ISO 8601 timestamp with timezone (`date -Iseconds`).
5. Set `status` to `"active"`.
6. Build YAML entry via `yq` and append to `todos.yaml` array.
7. If `todos.yaml` does not exist, create it with an empty array `[]` first.
8. Run `cmd_validate` on the result. If validation fails, remove the appended entry and die.
9. Print the generated `id` to stdout.

**Slug generation function:**

```sh
slugify() {
  printf '%s' "$1" \
    | tr '[:upper:]' '[:lower:]' \
    | sed 's/[^a-z0-9]/-/g; s/--*/-/g; s/^-//; s/-$//' \
    | cut -c1-50 \
    | sed 's/-$//'
}
```

**Exit codes:** 0 = success, 1 = error (missing flags, validation failure, write error).

**Output:** The generated TODO `id` on stdout. Errors on stderr.

### 3.2 `alm extract`

**Purpose:** Extract TODO candidates from row artifacts. Absorbs `scripts/extract-todo-candidates.sh`.

**Interface:**

```
alm extract [<row-name>]
```

- If `<row-name>` omitted, extract from all rows under `$ROWS_DIR/`.
- If `<row-name>` given, extract from that single row.

**Sources (per row):**

1. **`summary.md`** — Lines under `## Open Questions` heading (between it and next `##` or EOF).
   - Each non-blank line becomes a candidate.
   - `source`: `"summary-open-questions"`
   - `title`: truncated to 80 chars with `...` suffix if needed.
   - `context`: `"Open question from {row} row"`
   - `raw_content`: the full line text.
   - `source_file`: `.furrow/rows/{name}/summary.md`

2. **`learnings.jsonl`** — Lines where `category == "pitfall"` and `promoted == false`.
   - `source`: `"learnings-pitfall"`
   - `title`: `.content` truncated to 80 chars.
   - `context`: `.context` field or fallback `"Pitfall from {row} row"`.
   - `raw_content`: full JSON stringified.
   - `source_file`: `.furrow/rows/{name}/learnings.jsonl`

3. **`reviews/*.json`** — `.phase_b.dimensions[]` entries where `verdict != "pass"`.
   - `source`: `"review-finding"`
   - `title`: `"Review finding: {name} ({verdict})"`
   - `context`: `.evidence` or fallback.
   - `raw_content`: dimension object + deliverable stringified.
   - `source_file`: `.furrow/rows/{name}/reviews/{filename}`

**Output:** JSON array on stdout. Empty array `[]` if no candidates found.

**Implementation notes:**
- Uses temp directory with trap cleanup (same pattern as current script).
- Processes sources sequentially, collects into temp JSON files, merges with `jq -s 'add // []'`.
- Malformed JSONL lines and unparseable review files are silently skipped (same as current behavior).

**Exit codes:** 0 = success (including zero candidates), 1 = error (row not found, jq failure).

### 3.3 `alm list`

**Purpose:** List TODOs from `todos.yaml` with optional filtering.

**Interface:**

```
alm list [--seed] [--status S] [--json]
```

**Flags:**

| Flag | Description |
|------|-------------|
| `--seed` | Show `seed_id` column; for each linked seed, call `sds show <seed_id>` to fetch seed status |
| `--status S` | Filter by status: `active`, `done`, `blocked`, `deferred` (default: show all) |
| `--json` | Output raw JSON instead of formatted table |

**Table format (default):**

```
ID                    STATUS   URGENCY  TITLE
rename-to-furrow      active   high     Rename harness to furrow
add-gate-tests        done     medium   Add gate integration tests
```

With `--seed`:

```
ID                    STATUS   URGENCY  SEED             SEED STATUS   TITLE
rename-to-furrow      active   high     rename-furrow    implementing  Rename harness to furrow
add-gate-tests        done     medium   -                -             Add gate integration tests
```

**Behavior:**

1. Read `todos.yaml` via `yq -o=json`.
2. Apply `--status` filter if provided (jq `select`).
3. If `--json`, pipe filtered array to stdout and exit.
4. Format as aligned columns using `printf` or `column -t`.
5. If `--seed`, for each entry with a non-null `seed_id`, run `sds show <seed_id> --json 2>/dev/null` and extract `.status`. Display `-` if `seed_id` is absent or `sds show` fails.

**Exit codes:** 0 = success, 1 = error (file not found, parse error).

### 3.4 `alm show`

**Purpose:** Display full detail of a single TODO.

**Interface:**

```
alm show <id>
```

**Output format:**

```
TODO: rename-to-furrow
Status: active
Title: Rename harness to furrow
Context:
  The project needs a distinct identity separate from the generic harness name.
Work Needed:
  1. Update all path references
  2. Rename directory
Risks:
  Symlink breakage during transition
References:
  - commands/lib/step-transition.sh
  - scripts/install.sh
Source: manual
Created: 2026-03-15T10:30:00+00:00
Updated: 2026-04-01T14:22:00+00:00
Depends On: namespace-cleanup
Files Touched: commands/*, scripts/*
```

If `seed_id` is present, append seed info:

```
Seed: rename-furrow
Seed Status: implementing  (via sds show)
```

**Seed lookup:** Run `sds show <seed_id> --json 2>/dev/null`. If `sds` is not available or fails, print `Seed Status: (unknown)`.

**Exit codes:** 0 = success, 1 = not found or error.

### 3.5 `alm triage`

**Purpose:** Run dependency graph analysis and generate `roadmap.yaml`. Absorbs `scripts/triage-todos.sh`.

**Interface:**

```
alm triage [--full]
```

**Flags:**

| Flag | Description |
|------|-------------|
| `--full` | Regenerate all phases from scratch; default preserves completed phases |

**Behavior:**

1. Check `todos.yaml` exists; die with exit 2 if not found.
2. Run `cmd_validate` on `todos.yaml`; die with exit 3 if invalid.
3. Convert `todos.yaml` to JSON via `yq -o=json`.
4. Run embedded Python for graph analysis (Kahn's algorithm):
   - Filter active TODOs (status not in `done`, `deferred`).
   - Validate dependency references (dangling dep = exit 3).
   - Topological sort with wave assignment: `wave[node] = max(wave[dep] for dep in deps) + 1`.
   - Cycle detection: if `len(topo_order) != len(ids)`, report cycle and exit 3.
   - Build wave groups.
5. Detect file conflicts via jq (same glob-to-regex overlap check as current `triage-todos.sh`):
   - For each wave, check all TODO pairs for `files_touched` overlap.
   - Produce conflict zone records.
6. If `--full` is NOT set and `roadmap.yaml` exists:
   - Read existing `roadmap.yaml`.
   - Preserve phases where `status == "done"`.
   - Merge new graph data into remaining phases.
7. Write `roadmap.yaml` to `$ALM_ROADMAP`.
8. Output JSON analysis to stdout (same structure as current `triage-todos.sh` for backward compatibility).

**Embedded Python block:**

The Python code is passed via heredoc to `python3 -c`. It receives the JSON todos as `sys.argv[1]`. Stderr is captured to a temp file for error reporting. This is the same pattern as the current `triage-todos.sh`.

**roadmap.yaml generation:** After the Python+jq analysis produces the JSON graph, convert it to YAML via `jq` piped to `yq -P` (or construct YAML directly with `yq`), adding:
- `schema_version: "1.0"`
- `metadata` block with `generated_at`, `total_phases`, `completed_phases`
- `dependency_graph` with `nodes`, `edges`, `waves`
- `conflict_zones` array
- `phases` array (see section 5 for schema)
- `deferred` array (TODOs with status `deferred`)
- `handoff` template

**Exit codes:**

| Code | Meaning |
|------|---------|
| 0 | Success |
| 1 | Usage/dependency error |
| 2 | `todos.yaml` not found |
| 3 | Validation error (schema, dangling deps, cycles) |

### 3.6 `alm next`

**Purpose:** Generate handoff prompts for the next phase of work from `roadmap.yaml`.

**Interface:**

```
alm next [--phase N]
```

**Flags:**

| Flag | Description |
|------|-------------|
| `--phase N` | Target a specific phase number (default: first phase with status `in_progress` or `planned`) |

**Behavior:**

1. Read `roadmap.yaml`; die if not found.
2. Determine target phase:
   - If `--phase N` given, select that phase.
   - Otherwise, find the first phase where `status != "done"`.
3. For each work unit in the target phase, generate a handoff prompt from the `handoff.template`:
   ```
   Start with: `/work {branch_name} — {description}`
   Source TODOs: {todo_ids}
   Key files: {key_files}
   ```
4. Output handoff prompts to stdout, one per work unit, separated by blank lines.

**Read-only:** This command makes no mutations to any file.

**Exit codes:** 0 = success, 1 = error (file not found, phase not found).

---

### 3.7 `alm validate`

**Purpose:** Validate `todos.yaml` against schema and cross-field rules. Absorbs `scripts/validate-todos.sh`.

**Interface:**

```
alm validate [<path>]
```

- Default `<path>`: `$ALM_TODOS` (`.furrow/almanac/todos.yaml`)

**Behavior:**

1. Check file exists; die if not.
2. Locate schema at `$SCHEMA_FILE` (`adapters/shared/schemas/todos.schema.yaml`); die if not found.
3. **Schema validation** (requires `yq` + `python3`):
   - Convert both YAML files to JSON via `yq -o=json`.
   - Run `jsonschema` validation via embedded Python (`Draft202012Validator` with `Draft7Validator` fallback).
   - Collect all schema errors with their JSON paths.
4. **Cross-field validation:**
   - **Unique IDs:** Extract all `id` values via `yq -r '.[].id'`, check for duplicates via `sort | uniq -d`.
   - **Dependency references:** For each entry, verify all `depends_on` IDs exist in the file.
   - **Slug format:** Verify all IDs match `^[a-z][a-z0-9]*(-[a-z0-9]+)*$` (redundant with schema but belt-and-suspenders for cross-field context).
5. **Graceful degradation:** If `python3` is missing, skip schema validation with a warning. If `yq` is missing, skip all validation with a warning and exit 0.
6. Report all errors before exiting (do not fail on first error).

**Output:**

- On success: `"todos.yaml is valid"` to stdout.
- On failure: Error lines to stderr, one per issue.

**Exit codes:** 0 = valid, 1 = invalid.

### 3.8 `alm render`

**Purpose:** Generate human-readable Markdown from `roadmap.yaml`.

**Interface:**

```
alm render [--output PATH]
```

**Flags:**

| Flag | Description |
|------|-------------|
| `--output PATH` | Write Markdown to file at PATH (default: stdout) |

**Behavior:**

1. Read `roadmap.yaml`; die if not found.
2. Convert to JSON via `yq -o=json`.
3. Render Markdown sections using the template structure from `templates/roadmap.md.tmpl`:
   - **Header:** Title, updated date, status summary (completed/total phases).
   - **Dependency DAG:** ASCII graph built from `dependency_graph.edges` (simple indented format, not full graphviz).
   - **Legend:** Static notation key.
   - **Conflict Zones:** Table from `conflict_zones` array.
   - **Phases:** One section per phase with status, rationale, work units (branch name, description, TODOs, key files, conflict risk).
   - **Worktree Commands:** Shell snippets for launching parallel worktrees per phase.
4. The rendering order follows `templates/roadmap-sections.yaml` section ordering.
5. If `--output` given, write to that path. Otherwise, print to stdout.

**On-demand only:** This command is not called automatically by `alm triage`. The YAML is the source of truth; Markdown is a convenience view.

**Exit codes:** 0 = success, 1 = error.

## 5. `roadmap.yaml` Schema

### 5.1 Top-Level Structure

```yaml
schema_version: "1.0"

metadata:
  project: string               # Project name (e.g., "furrow")
  generated_at: string           # ISO 8601 timestamp
  total_phases: integer          # Total number of phases
  completed_phases: integer      # Phases with status "done"

dependency_graph:
  nodes:                         # All active TODO nodes
    - id: string                 # TODO id (kebab-case)
      label: string              # Human-readable label (from title)
      phase: integer             # Assigned phase number
      status: string             # active | done | blocked
  edges:                         # Dependency edges
    - from: string               # Source TODO id
      to: string                 # Target TODO id
      kind: string               # "hard" (explicit depends_on) | "inferred" (file conflict)
      reason: string             # Why this edge exists
  waves:                         # Topological wave groups
    - wave: integer              # Wave number (1-based)
      work_units: [string]       # TODO ids in this wave

conflict_zones:                  # File overlap warnings
  - phase: integer               # Phase where conflict occurs
    file_pattern: string         # The overlapping file/glob
    work_units: [string]         # TODO ids that overlap
    severity: string             # "low" | "medium" | "high"
    mitigation: string           # How to handle (e.g., "sequential ordering")

phases:                          # Ordered phase list
  - number: integer              # Phase number (1-based)
    title: string                # Phase title
    status: string               # "done" | "in_progress" | "planned"
    rationale: string            # Why this phase exists / ordering rationale
    work_units:                  # Work units within this phase
      - index: integer           # Position within phase (1-based)
        branch_name: string      # Git branch name for this work unit
        description: string      # What this work unit does
        todos: [string]          # Source TODO ids driving this work unit
        depends_on: [string]     # Other work unit branch_names this depends on
        key_files: [string]      # Primary files/globs this work unit touches
        conflict_risk: string    # "none" | "low" | "medium" | "high"
        completed_at: string     # ISO 8601 timestamp (null if not done)

deferred:                        # TODOs with status "deferred"
  - id: string                  # TODO id
    title: string                # TODO title
    reason: string               # Why deferred

handoff:
  template: string               # Handoff prompt template with {branch_name},
                                 # {description}, {todo_ids}, {key_files} placeholders
```

### 5.2 Phase Status Rules

- `"done"`: All work units have `completed_at` set. Preserved across `alm triage` runs (unless `--full`).
- `"in_progress"`: At least one work unit completed, at least one not.
- `"planned"`: No work units completed.

### 5.3 Wave-to-Phase Mapping

Waves from the dependency graph map to phases. The agent layer (Claude Code `/furrow:triage` command) is responsible for grouping waves into phases and assigning phase titles/rationale. The `alm triage` command produces the raw graph data; phase grouping logic uses wave numbers as input but may merge or split waves based on thematic coherence.

## 12. Testing Contract

`test-alm.sh` should cover:

1. **`alm add`**: creates entry, generates valid slug, handles collisions, validates post-write, rejects missing required flags.
2. **`alm extract`**: extracts from each source type (summary, learnings, reviews), handles missing files, outputs valid JSON, handles all-rows mode.
3. **`alm list`**: default output, `--status` filter, `--json` output, `--seed` with and without `sds` available.
4. **`alm show`**: existing id, non-existent id, seed linkage display.
5. **`alm triage`**: wave assignment, cycle detection, dangling dep detection, conflict detection, `--full` regeneration, phase preservation.
6. **`alm next`**: handoff prompt generation, `--phase` targeting, missing roadmap error.
7. **`alm validate`**: valid file, schema violations, duplicate IDs, dangling deps, missing file.
8. **`alm render`**: Markdown output correctness, `--output` file write, missing roadmap error.
9. **Edge cases**: empty `todos.yaml` (`[]`), single entry, no active TODOs, all deferred.
