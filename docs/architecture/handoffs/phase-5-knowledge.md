## Settled Decisions

All Phase 1-4 decisions are settled. Key knowledge/ecosystem decisions from research:

- Design for deletion: every component has an entry in `_rationale.yaml`
- Component deletion testing: periodically remove component, run evals, delete if evals pass
- Calibration is ongoing, not a one-time setup
- Agents can propose new tools/skills but human reviews before promotion
- Context promotes between tiers (work -> ambient) with human or eval gating
- MCP as the standard integration surface
- File-based artifact storage

## Deliverables

These 3 specs can be drafted in parallel.

### Spec 16: Artifacts (`docs/architecture/artifacts.md`)

**What**: The full lifecycle of every artifact type Furrow produces.

**Must include for each artifact type**:
- Format and structure
- Where it's stored (file path convention)
- Who produces it (human, agent, hook, eval)
- Who consumes it
- Lifecycle (creation -> active use -> archive -> deletion)
- How it moves between context tiers
- Entry in `_rationale.yaml`

**Artifact types to cover**:

1. **Work output artifacts**: Design docs, decision records, architecture notes produced during ideation and execution. Persist as work-tier context.

2. **Handoff prompts**: The highest-quality-bar artifact type — primary session continuity mechanism. Auto-generated by hook, consistent format, machine-parseable sections. More important than conversation history. Produced at every significant boundary. Specify exact format, sections, size target.

3. **Review artifacts / evidence packages**: The evidence package for trust levels 2-3. What it contains (eval results, work summary, git diff, trace highlights). Format. How it's presented to the reviewer. How it maps to the PR.

4. **Findings**: First-class entities with lifecycle.
   - Append-only storage, permanent IDs
   - Severity-based blocking (critical blocks advancement, important must be addressed or tracked)
   - Status tracking: OPEN -> FIXED / DEFERRED / WONTFIX
   - Findings feed back into self-improvement (recurring patterns trigger harness changes)
   - Storage format and location

5. **Specs as contracts**: Immutable once approved. Referenced by deliverables during execution. Cross-cutting specs (shared patterns) distinguished from component specs. How immutability is enforced.

6. **Futures**: Deferred enhancements discovered during work. Captured with rationale, linked to work that surfaced them. Available as context for related future work. Format and storage.

7. **Reusable artifacts**: Eval specs, team templates, specialist prompts, patterns that accumulate across work units. How they're indexed, versioned, and discovered.

**Cross-cutting concerns**:
- Artifact cleanup: when work is archived, what's retained and what's discarded?
- Artifact discovery: how does an agent find relevant artifacts from prior work?
- Tier promotion: how does a work-level artifact become a project-level convention?

### Spec 18: Self-Improvement (`docs/architecture/self-improvement.md`)

**What**: How Furrow evolves from its own operational data.

**Must include**:

- **Eval feedback loop**: How accumulated eval results identify patterns.
  - Frequent failures on specific behavior types -> which harness component needs improvement?
  - Aggregation method (per-behavior failure rates, trend detection)
  - Who reviews the patterns? (Automated detection, human review of recommendations)

- **Component deletion testing**: The research proposes "remove component, run evals, see what breaks."
  - Trigger: When is deletion testing run? (Scheduled? After model upgrades? Manual?)
  - Scope: Test one component at a time? Combinations?
  - Process: Remove component -> run full eval suite -> compare results -> human decides
  - Automation: Can this be an autonomous work type? (Meta: harness testing itself)
  - What counts as "evals still pass"? (Same pass rate? Within margin? All green?)

- **Prompt evolution**: How eval results feed back into prompts.
  - Specialist prompts that produce low eval scores -> what changes?
  - Evaluator criteria that produce low calibration agreement -> how are they updated?
  - Harness conventions that agents frequently violate -> structural enforcement or prompt improvement?
  - Version tracking for prompts (which version produced which results)

- **Calibration tracking**: How LLM-judge calibration data accumulates.
  - Storage format and location
  - What triggers recalibration (agreement rate drops below threshold)
  - Recalibration workflow (sample outputs, surface disagreements, update prompts, re-test)
  - Per-behavior calibration (not global — different behaviors may need different calibration)

- **Self-evolving scaffolds**: Controlled version of agents proposing new tools/skills.
  - Agent observes recurring pattern -> proposes new eval/skill/hook
  - Proposal format (what it is, why it's needed, evidence from operational data)
  - Human review gate (proposals never auto-promoted)
  - Testing: proposed component must have its own eval before promotion

- **Finding-driven prevention**: When findings recur, propose structural prevention.
  - Detection: same finding type appearing across multiple work units
  - Response: propose new hook, eval, or convention (not just fix the symptom)
  - Example: if agents repeatedly violate file ownership, propose a structural enforcement hook

- **Shrinkage over time**: How Furrow tracks its own trajectory toward getting smaller.
  - Component count over time
  - Deletion test results (what's been safely deleted)
  - Model capability tracking (which model limitations are the components encoding? are those limitations still present?)

### Spec 19: Integrations (`docs/architecture/integrations.md`)

**What**: How Furrow connects to external systems.

**Must include**:

- **MCP servers**: How Furrow uses MCP for persistent knowledge and tool extension.
  - Memory servers (work-log for persistent work history, personal-agent for preferences)
  - Code intelligence (Serena for semantic navigation — already in use)
  - Custom MCP servers for harness-specific tools
  - How MCP tools are provisioned per work type (skill injection matrix)

- **CI/CD integration**:
  - Do evals run in CI? (Deterministic evals yes, LLM-judge evals — cost/speed tradeoff)
  - Does CI trigger autonomous work? (Test failure -> bug fix work unit)
  - How does PR creation integrate with CI pipelines?
  - Eval result reporting in CI (status checks, PR comments)

- **Issue trackers**:
  - How work definitions map to issues/tickets
  - Bidirectional sync or one-way? (Recommendation: one-way, work def -> issue, keep minimal)
  - Issue creation from work completion (auto-create tickets for futures/findings)
  - Keep at beans-level simplicity, not Jira-level complexity

- **Team and organizational tools**:
  - Notion for documentation and knowledge management
  - Claude desktop/cowork for collaborative sessions
  - Notification channels (desktop notifications via notify-send, email, chat)

- **Per-project configuration**:
  - Stack metadata (language, framework, build commands, test commands)
  - Where this lives (CLAUDE.md? separate config file? work definition?)
  - How it propagates to all agents automatically
  - Single source of truth — no duplication

- **Integration extensibility**:
  - MCP as the standard integration surface
  - How new integrations are added (new MCP server, register in config)
  - Integration health checks (verify external systems are reachable)
  - Graceful degradation (what happens when an integration is down?)
